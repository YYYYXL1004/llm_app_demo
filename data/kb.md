# 本地小模型部署思路

本地或服务器部署小模型的目标，是在隐私、成本、可控性和演示稳定性之间取得平衡。课堂 demo 推荐从 7B 到 14B 级别模型开始，根据显存选择量化版本。Ollama 适合教学，因为它统一了模型拉取、运行和 HTTP API。

部署时先确认三件事：模型是否能跑起来、API 是否能被应用访问、访问权限是否安全。公网不要直接暴露推理端口，至少使用防火墙、SSH tunnel 或带鉴权的反向代理。

# Prompt 基础

Prompt 的基本结构可以拆成角色、任务、上下文、约束和输出格式。角色告诉模型以什么身份回答，任务说明要解决什么问题，上下文提供事实，约束限制不能做什么，输出格式让结果可解析。

一个稳定的课堂 Prompt 示例：你是课程助教；只基于给定资料回答；不知道就说不知道；答案分三点；不要编造命令或结果。

# RAG 流程

RAG 是 Retrieval-Augmented Generation。标准流程是：准备文档，切分 chunk，建立索引，用户提问，检索相关 chunk，把检索结果放入 Prompt，模型基于上下文回答，最后返回答案和引用来源。

RAG 的主要价值是减少幻觉、接入私有知识、让答案可追溯。RAG 的常见问题包括 chunk 太大、检索词不匹配、召回内容不完整、上下文太长、模型没有遵守资料边界。

# Tool Calling

Tool Calling 是让模型输出结构化工具调用请求，再由程序执行真实函数。模型不应该自己假装查数据库、算精确结果或访问外部系统。正确流程是：给模型工具 schema，模型选择工具和参数，程序执行工具，工具结果回填给模型，模型生成最终回答。

适合做成工具的能力包括搜索、数据库查询、计算器、时间查询、文件处理、订单查询和业务系统 API。工具必须有明确输入输出、权限控制、超时、错误处理和日志。

# LLM 常见失败案例

常见失败包括幻觉、过度自信、上下文遗漏、工具参数错误、循环调用工具、把用户问题误解为指令、泄露系统提示词、输出格式不稳定、长对话状态污染。

工程上常用的处理方式包括：低温度、结构化输出、检索引用、工具执行校验、重试、追问、fallback、人工确认和审计日志。

# 如何把 Day 3 工具接入 LLM

把 Day 3 工具接入 LLM 的步骤是：先把工具封装成普通 Python 函数；定义 JSON Schema 描述函数名、用途和参数；把 schema 传给模型；接收模型返回的 tool call；在后端执行函数；把函数结果作为 tool message 回填给模型；最后让模型组织自然语言回答。

注意不要让模型直接执行危险操作。删除、支付、发邮件、改数据库这类动作要加确认节点。课堂 demo 可以先接入只读工具，例如搜索、计算、查时间、查课程资料。

# Chatbot、Workflow、Agent 的区别

Chatbot 主要是对话界面，输入一句话，模型回答一句话。Workflow 是确定性流程，步骤由程序控制，比如先检索再总结。Agent 是让模型参与决策，例如判断是否需要调用工具、是否追问、是否重试、下一步走哪条路径。

产品化时不要一开始就做完全自主 Agent。可控的方式是先做 Workflow，再在少数节点引入模型决策。

# 工具选择、状态管理、追问、重试、fallback

工具选择要看任务是否需要外部事实、精确计算或业务系统。状态管理要保存用户目标、历史消息、工具调用结果和未完成任务。追问用于补齐关键参数。重试用于处理临时失败，但要限制次数。fallback 用于模型或工具不可用时给出可执行替代路径。

一个可靠的 Agent 节点通常包含：输入校验、模型判断、工具调用、结果校验、失败重试、最终响应。

# Gradio 页面结构

Gradio 适合快速做教学 demo。`Blocks` 适合自定义布局，`ChatInterface` 适合快速创建聊天页面。一个最小聊天函数接收 message 和 history，返回字符串。通过 additional_inputs 可以把模式、temperature、top_k、system prompt 等控件传入聊天函数。

课堂上建议用一个页面展示多个模式，而不是拆成多个项目。这样同学能看到 Chat、RAG、Tool Calling 和 Agent Fallback 的差异。
